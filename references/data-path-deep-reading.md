# FPGA Data Path Deep Reading SOP

> 目标：在已经识别工程主数据链路之后，选择其中一条链路进行细读，回答“数据是什么、从哪来、怎么变、按什么时钟走、在哪里被缓存、最后到哪里去、如何验证”。

这个 SOP 不替代 `$fpga-cowork` 的全工程阅读流程。它用于第二阶段：从工程地图进入一条具体数据通路。

## 0. 核心原则

### 0.1 数据守恒

读数据通路时，先追踪数据本身，不先追踪所有控制信号。

每一级都要问：

| 问题 | 目的 |
|---|---|
| 输入数据是谁？ | 找生产者 |
| 输出数据是谁？ | 找消费者 |
| 数据宽度是否变化？ | 判断打包、截位、扩展、复用 |
| 数据语义是否变化？ | 判断采样点、像素、包、命令、坐标等含义 |
| valid/ready/wr_en/rd_en 条件是什么？ | 判断数据什么时候算有效 |

如果某一级只改变控制，不改变业务数据，把它标记为支撑节点，不要让它抢走主线。

### 0.2 边界优先

先确认通路的两个端点：

| 端点 | 要确认的内容 |
|---|---|
| Source | 数据最早在哪里被外部输入、寄存器生成、ROM/FIFO 读出或算法产生 |
| Sink | 数据最终写到外部管脚、DDR、Ethernet、SFP、DAC/ADC、BRAM 或另一个业务模块 |

没有端点，不读内部实现。

### 0.3 时钟是数据的时间坐标

每段通路必须标注时钟域：

| 类型 | 读法 |
|---|---|
| 同时钟组合/寄存 | 看 `always @(posedge clk)` 或连续赋值 |
| 异步 FIFO | 标出写时钟、读时钟、数据宽度、深度和满空策略 |
| 同步器 | 标出同步的是控制还是数据 |
| 外设时钟 | 标出是否由外部芯片/IP 决定，避免凭 RTL 推断模拟速率 |

跨时钟处要单独列为风险点或验证点。

### 0.4 控制约束数据，而不是替代数据

控制信号要按“它约束哪段数据”来读。

| 控制类型 | 典型信号 | 关注点 |
|---|---|---|
| 模式选择 | `mode`、`state`、`sel` | 选择哪一路数据或哪套参数 |
| 采样节奏 | `sample`、`interval`、`cnt` | 决定数据产生频率/步进 |
| 传输握手 | `valid`、`ready`、`wr_en`、`rd_en` | 决定数据是否进入下一段 |
| 资源仲裁 | `grant`、`busy`、`prog_full` | 决定是否丢数据、暂停或切换 |
| 复位/启动 | `rstn`、`start`、`trigger` | 决定数据生命周期 |

### 0.5 先建立合同，再读实现

对每个模块先写接口合同：

```text
module:
clock/reset:
input data:
output data:
control in:
control out:
storage:
mode:
contract:
```

合同写清后，再进入 `always` 块或状态机。

### 0.6 反向读，正向验证

单条数据通路的第一遍阅读方向不一定等于数据真实生产方向。

| 通路类型 | 第一遍推荐阅读方向 | 原因 |
|---|---|---|
| 输出链路 | 从外部 Sink/管脚/最后一级锁存反向追到 Source | 最终边界最明确，能先拿到上游必须满足的数据合同 |
| 输入链路 | 从外部 Source/采样入口正向追到第一处稳定缓存或消费点 | 输入数据的物理入口最明确 |
| DDR/BRAM 中转链路 | 从业务端点和读写方向同时夹逼 | 存储本身只是中转，不能把“写 DDR”误当成业务终点 |
| Mux/多来源链路 | 先找到汇流后的统一合同，再逐个读来源分支 | 避免一开始陷入某个支路算法 |

反向读的目标是建立“下游到底要什么”；之后必须正向验证“上游是否真的按这个合同生产”。

### 0.7 默认产物是代码阅读指导手册

当用户说“精读某条数据通路”“读下某条通路”“复用/检验这个技能”或没有明确要求总结型文档时，默认输出“代码阅读指导手册”，而不是结论总结。总结型细读文档只在用户明确要求“总结”“概览”“只要结论”时使用。

代码阅读指导手册的验收标准不是“我已经读懂并总结出来”，而是“后续工程师能按文档重新打开源码，复现同一条阅读路径”。如果文档不能指导读者逐文件阅读，就算链路结论正确，也视为不合格。

阅读手册要告诉读者：

| 内容 | 作用 |
|---|---|
| 打开哪个文件 | 限定本轮范围 |
| 搜什么关键字 | 避免全文阅读 |
| 哪些信号是主线 | 保护注意力 |
| 哪些先忽略 | 防止支撑逻辑抢走主链 |
| 按什么顺序读局部代码 | 形成可复现路径 |
| 读完应能回答什么 | 给每一轮设置退出条件 |

每个主要“打开文件 / Pass”章节都必须包含这些字段：

```markdown
本步目标：
搜索入口：
本模块相关信号：
先忽略：
代码阅读顺序：
读完应能回答：
下一步：
```

其中“先忽略”和“代码阅读顺序”不可省略。复杂 RTL 文件里，保护读者注意力和限定局部阅读顺序，是这个子技能的核心能力。

### 0.8 交付物决策表

先判断用户要的是哪一种产物。不要把两种形态混在一起。

| 用户意图 | 应交付 | 不要交付 |
|---|---|---|
| “读一下这个工程”“生成 FPGA_PROJECT_GUIDE”“先梳理有哪些数据通路” | 全工程数据链路地图 / 项目指南 | 单条通路源码细读 |
| “精读某条通路”“深读 ADC/DAC/DDR 路径”“按技能第二步读这条链” | 代码阅读指导手册 | 架构摘要或证据清单 |
| “总结这条通路”“只要结论”“概览一下” | 总结型细读文档 | 过长的逐文件阅读手册 |
| “评审/修订已有通路文档” | 先按本 SOP 做缺口检查，再重写缺失章节 | 只评价好坏不落到结构问题 |

如果用户没有明确说“总结/概览/只要结论”，选中通路的默认产物就是代码阅读指导手册。

### 0.9 常见失败模式

这些产物即使结论基本正确，也不能算合格的 selected-path deep read：

| 失败模式 | 为什么不合格 | 修正方式 |
|---|---|---|
| 每个 Pass 只有“关键证据”表 | 读者不知道打开文件后搜什么、先忽略什么、按什么顺序读 | 给每个主要 Pass 补齐七个必填字段 |
| 按模块功能总结全链路 | 这是架构摘要，不是可复现阅读路径 | 改成“第 N 步：打开 `xxx.v`” |
| 复杂文件被压成一行结论 | 容易漏掉状态机、FIFO、mux 的局部读法 | 限定搜索入口和局部阅读顺序 |
| DDR/Ethernet/FIFO 被当成主链路终点 | 存储/接口可能只是中转，不代表业务语义结束 | 用 Source/Sink 和业务数据语义重新判定主链 |
| 控制、复位、debug 与真实数据混写 | 主线注意力被支撑逻辑抢走 | 将支撑项放进“先忽略”“CDC/缓存”“控制约束”章节 |
| 没有说明分支如何回到统一合同 | 多来源路径读散了，无法形成通路闭环 | 先找 mux 后的统一合同，再读每个来源 |
| 没有发布前自检 | 容易把第一阶段项目指南格式误用到第二阶段 | 用本文的必填字段或校验脚本检查 |

## 1. 细读流程

### Step 1：定义本次通路切片

只读一条链，不泛化到全工程。

输出：

| 项目 | 内容 |
|---|---|
| 通路名称 | 例如 DAC 输出链路 |
| Source | 例如上位机寄存器参数/扫描生成器 |
| Sink | 例如 `dac_p1d/dac_p2d` 到 AD9747 |
| 主数据 | 例如 2 路 16-bit DAC 点 |
| 主时钟 | 例如 `eth_clk` 生成参数，`dac_dco` 输出数据 |
| 第一遍阅读方向 | 例如从 `dac_p1d/dac_p2d` 反向追到 `gen_data_mux` 和参数源 |
| 输出产物 | 默认代码阅读指导手册；总结型文档仅在用户明确要求时使用 |
| 验收基准 | 后续读者能按文档逐文件打开源码、搜索关键词、跳过无关支撑逻辑并复现结论 |
| 暂不读 | 例如 ADC 图像、远程升级、调试 ILA |

### Step 2：列问题清单

用问题驱动搜索，不全文读。

| 问题 | 推荐搜索 |
|---|---|
| 谁最终驱动输出端口？ | `rg -n -C 5 "dac_p1d|dac_p2d|DAX_DATA|DAY_DATA"` |
| 这个数据由哪个模块产生？ | `rg -n -C 5 "DAX_DATA|DAY_DATA|DA1_DATA|DA2_DATA"` |
| 数据在哪个时钟域产生？ | `rg -n -C 5 "dac_dco|wr_clk|rd_clk|posedge"` |
| 参数从哪里来？ | `rg -n -C 5 "dac_sample|dacx_strat|scan_mode|wr_reg_addr"` |
| 哪些模式改变路径？ | `rg -n -C 5 "scan_mode|case|mill_state|sel"` |
| 哪里可能阻塞或丢数据？ | `rg -n -C 5 "prog_full|empty|full|busy|wr_en|rd_en"` |

每个问题只读命中窗口，必要时再扩大到 50-120 行。

同时把问题整理成阅读 Pass 表。每个 Pass 只解决一个窄问题，避免“打开一个文件顺便全读完”。

| Pass | 打开文件 | 本轮只解决的问题 | 退出条件 |
|---:|---|---|---|
| 1 | 顶层或边界文件 | 外部端点和内部主数据信号名是什么 | 能说清外部端口和内部信号的映射 |
| 2 | 端点包装层 | 数据从哪个子模块进出 | 能画出上一跳/下一跳 |
| 3 | 汇流/拆包/CDC 模块 | 统一数据合同是什么，在哪里跨时钟 | 能列出位宽、位段、时钟、读写条件 |
| 4 | 来源选择模块 | 哪些来源会写同一个合同 | 能列出 mode/state 到数据来源的关系 |
| 5+ | 各来源分支 | 每个来源如何生产合同 | 能回到统一合同，而不是发散成全工程阅读 |

Pass 表不是固定 5 步。它要跟随当前通路，但每一行都必须有明确的“本轮只解决的问题”。

常见通路的 Pass 种子如下。它们只是起点，实际文件名和顺序必须由工程证据决定。

| 通路类型 | 推荐第一遍阅读骨架 |
|---|---|
| 输入采集链路 | 外部 Source/采样管脚 -> 顶层裁位/时钟入口 -> 入口 mux -> 单路采集/平均 -> CDC FIFO -> 多路聚合/格式化 -> 存储或回传入口 |
| 输出驱动链路 | 外部 Sink/管脚 <- 顶层映射 <- 最后一级输出锁存 <- CDC/输出 FIFO <- 统一数据合同 <- mux/来源选择 <- 生产者分支 <- 控制寄存器 |
| DDR/BRAM 中转链路 | 写侧业务数据 -> 写格式化/地址/长度 -> 存储接口 -> 读请求/地址/长度 -> 读侧格式化 -> 最终业务消费者 |
| 多来源 mux 链路 | 先读 mux 输出合同 -> 选择条件 -> 各来源如何生产合同 -> 反压/冲突处理 -> 统一验证点 |
| 包化/回传链路 | 业务数据输入 -> 宽度转换/缓存 -> 包头/包计数 -> 发送握手 -> 外部接口 payload |

### Step 3：画一条最小通路图

先画文本图，不急着画漂亮图。

```text
Producer
  -> formatter / generator
  -> clock-domain bridge
  -> output formatter
  -> external endpoint
```

图里只放真实数据节点。控制节点挂在旁边，不放进主链路。

如果第一遍采用反向阅读，建议同时保留两张小图：

```text
真实生产方向：
Source -> generator/mux -> CDC/buffer -> output latch -> Sink

第一次阅读方向：
Sink <- output latch <- CDC/buffer <- mux/generator <- Source
```

并先写出关键名字和数据包位段：

| 项目 | 示例 |
|---|---|
| 内部主数据信号 | `DAX_DATA`、`DAY_DATA` |
| 统一数据合同 | `para_config_wr_en + para_config_data[33:0]` |
| 位段解释 | `{frame_end, adc_tri, x[15:0], y[15:0]}` |
| 最后一级锁存 | `DA1_DATA/DA2_DATA` |
| 外部端口映射 | `dac_p1d/dac_p2d` |

这一步的目的不是把所有逻辑讲完，而是给后续阅读建立最小心智模型。

### Step 4：逐段建立模块合同

每经过一个模块，记录：

| 模块 | 输入数据 | 输出数据 | 时钟 | 控制 | 缓冲 | 当前结论 |
|---|---|---|---|---|---|---|

优先阅读模块端口和实例连接。只有合同不清楚时，才进入内部状态机。

如果产物是阅读手册，每个模块/Pass 都按固定小节写。选中通路的主要文件不能只写“关键证据表”，必须写成可执行的阅读步骤：

```markdown
### 第 N 步：打开 `xxx.v`

本步目标：

搜索入口：

本模块相关信号：

先忽略：

代码阅读顺序：

读完应能回答：

下一步：
```

其中“先忽略”是必填项。复杂 FPGA 文件经常把 ADC、DAC、DDR、升级、debug、状态回读混在一起，明确忽略清单比解释所有信号更重要。

发布前自检：如果一个主要 Pass 缺少“本步目标 / 搜索入口 / 本模块相关信号 / 先忽略 / 代码阅读顺序 / 读完应能回答 / 下一步”中的任意项，不要把它当作合格的代码阅读指导手册。

如果你维护的是 Markdown 文件，优先运行：

```bash
python scripts/validate-deep-reading-guide.py docs/<path-guide>.md
```

脚本只做结构检查，不能替代工程判断；但如果脚本失败，通常说明文档已经偏成总结型产物，需要先补回逐文件阅读步骤。

### Step 5：识别数据形态变化

用“形态变化表”记录每一步。

| 位置 | 数据名 | 位宽 | 语义 | 有效条件 | 变化 |
|---|---|---|---|---|---|
| 寄存器输出 | `dac_sample` | 24 | DAC 点节奏参数 | 寄存器写 | 参数 |
| 参数流 | `para_config_data` | 34 | `{frame_end, adc_tri, x, y}` | `para_config_wr_en` | 打包 |
| DAC 点 | `DA1_DATA/DA2_DATA` | 16+16 | DAC X/Y 点 | FIFO 读 | 拆包 |

这张表是细读的核心产物。它必须独立成表，不能只散落在段落描述里。每次数据位宽、语义、有效条件或打包方式变化，都要回到这张表。

### Step 6：单独读 CDC 和缓存

所有 FIFO/CDC 必须单独列：

| 结构 | 写时钟 | 读时钟 | 数据 | 写条件 | 读条件 | 风险 |
|---|---|---|---|---|---|---|

重点看：

- FIFO 宽度是否和数据打包一致。
- `prog_full/full` 是否反馈到生产者。
- `empty` 时消费者是否保持旧数据或输出默认值。
- 复位是否跨域安全。

### Step 7：读模式和状态机

只读和该通路有关的状态。

| 模式/状态 | 数据来源 | 数据去向 | 特殊行为 |
|---|---|---|---|

判断规则：

- `case mode` 通常决定“哪一路数据进入主链路”。
- `state` 通常决定“什么时候开始/停止/保持”。
- `busy/full/empty` 通常决定“是否暂停或丢弃”。

读多来源汇流时，先确认汇流后的统一输出合同，再读来源分支：

| 支路类型 | 初读重点 | 暂不展开 |
|---|---|---|
| 普通生成器 | 是否输出统一合同、何时 `wr_en` | 复杂状态机每个分支 |
| 算法/图元生成器 | 最终点包位宽和 mux 选择条件 | 每种图元内部插值算法 |
| 单点/寄存器直写 | 寄存器字段如何直接组成数据包 | 无关寄存器回读 |
| 配置支路 | 是否只配置外设，不承载主数据 | SPI/I2C 每个 bit 的外设手册含义 |

判断支路读完的标准：能说明它如何回到主链路合同，而不是能讲完该支路内部所有算法。

### Step 8：读控制寄存器

把上位机或外部控制整理成寄存器表。

| 地址 | 字段 | 影响的数据段 | 说明 |
|---|---|---|---|

只记录影响本通路的字段。其他字段跳过或标记为支撑。

### Step 9：形成验证点

每条细读通路至少提出这些验证点：

| 验证点 | 观察信号 |
|---|---|
| Source 是否产生数据 | 参数/输入 valid、计数器 |
| CDC 是否正常 | FIFO full/empty、wr_en/rd_en |
| 模式切换是否正确 | mode/state、mux 输出 |
| Sink 是否收到数据 | 端口数据、valid、外设时钟 |
| 边界条件 | reset、stop、empty、full、mode 切换 |

验证信号要按组给出，便于仿真或 ILA 分批观察：

| 组 | 观察重点 |
|---|---|
| 出口组 | 外设时钟、内部最后一级数据、外部端口映射 |
| FIFO/CDC 组 | `wr_en/data`、`rd_en/dout`、full/empty/rst_busy |
| 输出锁存组 | 最后一级寄存器、同步标记、保持旧值/默认值行为 |
| 来源选择组 | mode/state、各来源 `wr_en/data`、mux 输出 |
| 控制入口组 | 影响本通路的寄存器字段、trigger/start/stop |

不要一次性把所有信号塞进 ILA。先看能证明链路是否跑通的最小组，再按风险扩展。

## 2. 读代码顺序

默认原则：先端点，再合同，再来源；先端口/实例，再局部实现。

输出链路推荐顺序：

1. 外部 Sink 和顶层实例连接：确认最终端口、内部主信号名、映射关系。
2. 最后一级消费者/输出锁存：确认下游真正需要的数据合同。
3. CDC/FIFO/拆包模块：确认统一数据包如何跨域、拆包、保持或丢弃。
4. 汇流/mux 模块：确认哪些来源能写入同一个合同。
5. 各生产者分支：按 mode/state 分支逐个确认如何生产合同。
6. 控制寄存器/参数源：最后读，只保留影响本通路的字段。
7. 外设配置/debug/ILA/IP 细节：主链读通后再看，作为支撑和风险确认。

输入链路推荐顺序：

1. 外部 Source 和采样入口：确认数据从哪里进入 FPGA。
2. 第一处锁存/解串/协议解析：确认原始数据如何变成内部合同。
3. CDC/FIFO/缓存：确认跨域、背压、满空策略。
4. 消费者或写入端：确认最终被谁使用、写到哪里。
5. 控制寄存器和 debug：最后挂到对应数据段。

Mux/多来源链路补充规则：

1. 先读 mux 输出合同，不先读每个来源内部。
2. 再读选择条件：mode/state/full/busy。
3. 最后按最常用或最关键支路展开来源。

不推荐顺序：

1. 从第一个 `always` 块开始读。
2. 先读所有寄存器地址。
3. 先读 IP 生成文件。
4. 先读 ILA/debug。
5. 把复位、时钟、配置线当成主数据线。
6. 在没有统一合同前展开所有算法分支。

## 3. 输出格式

### 3.1 总结型细读文档

仅在用户明确要求总结、概览或结论沉淀时使用。不要把它作为“精读某条通路”的默认交付：

```markdown
## [数据通路名称] 细读

### 1. 一句话结论
### 2. 最小通路图（真实生产方向 + 首次阅读方向）
### 3. 关键名字和统一数据合同
### 4. 模块合同表
### 5. 数据形态变化表
### 6. CDC/缓存表
### 7. 模式/状态表
### 8. 控制寄存器表
### 9. 验证点
### 10. 待确认和风险点
### 11. 关键证据索引
```

### 3.2 代码阅读指导手册

这是“精读某条通路”的默认交付形态。适合带人重新打开源码阅读。DAC 文档采用的是这种形态，也是本子技能的验收样例。

```markdown
# [数据通路名称] 代码阅读指导手册

## 0. 使用方法
## 1. 先建立最小心智模型
## 2. 第一步：打开 `[边界/顶层文件]`
## 3. 第二步：打开 `[下一跳模块]`
## N. 支撑模块，最后再看
## N+1. 上板或仿真时看哪些信号
## N+2. 待确认和风险点
## N+3. 关键证据索引
```

每个“打开文件”的章节必须包含：

| 小节 | 要写什么 |
|---|---|
| 本步目标 | 这一轮只解决的窄问题 |
| 搜索入口 | 关键字列表，不要求全文阅读 |
| 本模块相关信号 | 只列本通路相关信号 |
| 先忽略 | 明确哪些信号/模块暂时不读 |
| 代码阅读顺序 | 端口、实例、assign、always、状态机的局部顺序 |
| 读完应能回答 | 本轮退出条件 |
| 下一步 | 下一个文件或支路 |

写作要求：

- 每个主要章节用“第 N 步：打开 `xxx.v`”组织，而不是只用模块功能命名。
- “关键证据表”可以保留，但不能替代“搜索入口 / 本模块相关信号 / 先忽略 / 代码阅读顺序”。
- 每个分支必须回答“它如何回到统一主合同”。不能只列分支内部功能。
- 如果某个文件过大，只读和本 Pass 问题相关的端口、实例、assign、FIFO、状态机片段。
- 如果读到风险点，写在该 Pass 旁边，并在最终风险表中汇总。

### 3.3 证据索引格式

文档最后要保留证据索引，方便复核和后续修订：

| 证据 | 文件位置 |
|---|---|
| 顶层端口或外设入口 | `path/to/file.v:line` |
| 统一数据合同定义 | `path/to/file.v:line` |
| CDC/FIFO 例化 | `path/to/file.v:line` |
| mux 选择条件 | `path/to/file.v:line` |
| 控制寄存器写地址 | `path/to/file.v:line` |

证据索引不要求一开始完整，但发布前至少覆盖端点、统一合同、CDC/mux、关键控制入口和风险点。

## 5. 完成标准

一条数据通路算读完，必须能回答：

1. 数据从哪里来，到哪里去。
2. 每一级模块的合同是什么。
3. 数据位宽和语义在哪里变化。
4. 哪些控制信号改变数据路径或节奏。
5. 哪些 FIFO/CDC 是关键风险点。
6. 哪些寄存器字段能驱动这条通路变化。
7. 如何在仿真、ILA 或板级观测中验证这条通路。
8. 哪些结论仍然需要原理图、协议、IP GUI 或实测确认。
9. 如果交付阅读手册，后续读者是否知道每一步打开哪个文件、搜什么、先忽略什么、读完回答什么。
10. 关键结论是否都有 `path:line` 证据，不能只靠记忆或推断。

## 6. 与 Feature Development 的边界

Mode 2 读已有数据通路，产出可复用阅读理解。Mode 5 修改或新增功能，产出开发工作包。

如果细读后发现需要改 RTL、加寄存器、补 testbench、跑综合或做 ILA：

1. 在本阅读手册最后写“建议进入 Mode 5”。
2. 列出建议新建的 feature/unit、需求待确认项、风险和建议验证点。
3. 不要在 Mode 2 文档里继续写需求计划、实施进度或仿真 PASS 证据。
4. 已验证的新事实最终写入 `*_AS_BUILT.md`；阅读手册 `*_DEEP_READ.md` 不承担开发过程记录。
